# 欄位契約：訪客信息的 10 個欄位

訪客信息是一組**封閉的 10 欄位**。schema 未列出的 key 一律報錯，10 個欄位
全部必填，查無資料填半形 `-`。

`assets/profile.schema.json` 是唯一真相來源，`scripts/validate_profile.py`
在執行時讀它，兩支產檔腳本啟動時都會呼叫，**驗證失敗就不產檔**。

## 10 個欄位

| # | 欄位 | JSON key | 限制 |
|---|---|---|---|
| 1 | 姓名 | `name`（+ `name_en`） | 必填非空 |
| 2 | 性別 | `gender` | 必填 |
| 3 | 出生年月 | `birth` | 必填 |
| 4 | 生肖 | `zodiac` | 必填 |
| 5 | 籍貫 | `hometown` | 必填 |
| 6 | 聯繫方式 | `contact` | 必填 |
| 7 | 教育經歷 | `education` | 每列含畢業院校／專業／學歷／學位，四格皆必填；**最多 5 列** |
| 8 | 現任職位 | `positions` | **最多 3 個** |
| 9 | 主要經歷 | `career` | 每列含起止年月／工作單位／職位；**最多 10 筆** |
| 10 | 照片 | `photos` | 0～2 張 |

**xlsx 的儲存格位置刻意不跟著這個順序走。**表格裡 `B5` 仍是聯繫方式、`D5`
仍是籍貫，因為那是公司現行「個人信息登記表」範本的位置：跟著換會讓產出的
檔案偏離範本，而且 `xlsx_to_profile_json.py` 是按固定座標讀取的，換了之後
所有既有舊檔會被無聲讀錯（聯繫方式被當成籍貫）。欄位編號規範的是**這份表
單的欄位次序**，不等同 xlsx 的物理排版。改動前先讀 `xlsx-source-format.md`。

`name_en` 是欄位[1]的一部分，不是第 11 個欄位：寫進 xlsx 時併入 B3 成為
`中文名（English）`，讀回時再拆開。拆解只在括號內是拉丁字母時發生，所以
`王小明（範例）` 這種中文附註會原封不動留在 `name` 裡。

## 三個非欄位的 key

`timestamp`、`note`、`sources` 渲染在標頭與頁尾，**不是表格欄位**，所以不
違反「只有 10 欄位」：

- `timestamp` — 標頭右上的資料彙整時間。
- `note` — 頁尾備註。標註哪些內容是推估、照片是否非官方來源。寫法見
  `note-writing-guide.md`。
- `sources` — 頁尾資料來源清單，**至少 1 筆**。每項事實都要能回溯出處，
  這是本 skill 隱私與合規立場的基礎，不因欄位收斂而放寬。

## 查無資料一律填 `-`

半形 `-`，不是全形 `－`、不是破折號 `—`、不是「未知」「不詳」「N/A」
「TBD」，也不是留空或 `null`。驗證器會擋下所有其他寫法。

這條規則的用途是**可 grep**：任何人都能一眼數出這份檔案有幾格沒查到資料。
允許近似寫法等於廢掉這個約定。

> **注意：這與 2026-07 之前的約定相反。**舊版要求 `不詳（未公開）`，現在
> 該寫法會被驗證器拒絕。手邊若有舊的 profile.json，需要把
> `不詳（未公開）` 換成 `-` 才能重新產檔。從 xlsx 讀入時，
> `xlsx_to_profile_json.py` 的 `cell_value()` 會自動把各種空值寫法
> 正規化成 `-`，所以舊 xlsx 不需要手動處理。

## 硬限制（違反即報錯、不產檔）

| 規則 | 為什麼是硬的 |
|---|---|
| 未列於 schema 的 key | 欄位集合封閉，多一個就不是這份表 |
| 10 個欄位缺任一個 | 表單欄位不可省略，無資料也要填 `-` |
| `education` > 5 | 規格明訂上限 |
| `positions` > 3 | 規格明訂上限 |
| `career` > 10 | 規格明訂上限 |
| `photos` > 2 | 版型只排得下兩張 |
| `sources` < 1 | 事實必須可回溯 |
| 任何字串等於 `未知`／`不詳（未公開）`／`—`／`TBD` 等 | 空值只允許一種寫法 |

## 軟限制（超過只警告，仍會產檔）

**目前沒有任何欄位使用軟限制**——四個有數量上限的欄位（教育經歷 5、現任職位
3、主要經歷 10、照片 2）都是規格明訂的數字，因此全部是硬限制。

`x-soft-max` 的機制仍保留在 `validate_profile.py` 裡，供未來需要「提醒但不
擋下」的欄位使用：在 schema 的該欄位加上 `"x-soft-max": N` 即可，不需要改
程式碼。適用時機是**編輯性偏好**（例如「超過這個長度版面會很難讀」），而非
規格要求——硬擋一個規格沒禁止的情況，只會逼使用者去改 schema 或刪掉真實資料。

## 手動驗證

```bash
python3 scripts/validate_profile.py profile.json --target xlsx
python3 scripts/validate_profile.py profile.json --strict   # 警告也視為失敗
```

## 要修改限制時

改 `assets/profile.schema.json`，**不要**把規則寫死在 Python 裡——單一真相
來源正是這套設計的重點。唯一的例外是 JSON Schema 表達不了的空值寫法黑名單，
它寫在 `validate_profile.py` 裡並在該處註明原因。
